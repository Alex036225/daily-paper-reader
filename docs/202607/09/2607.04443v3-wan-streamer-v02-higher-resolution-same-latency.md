## 摘要
我们提出了 Wan-Streamer v0.2，这是对原生流式端到端音视频交互模型的延迟保持升级。v0.2 保持了 v0.1 的建模形式，但将交互输出流从 192×336 提升到 640×368，同时在 25 FPS 下保持约 200 毫秒的模型端信号到信号延迟。更高分辨率流支持场景扎根的中景镜头代理，其姿态、注视、手部、附近物体和局部场景布局在实时对话中保持清晰。为了在不增加用户感知延迟的前提下支持更大的视觉流，v0.2 将 think 器保留为单 GPU 低延迟路径，用于流式感知、简短的语言/状态 Transformer 传递以构建生成缓存，以及最终解码。执行器则成为多 GPU Ulysses 风格的上下文并行组，负责昂贵的下一单元潜在生成。每个执行器 rank 将传入的 K/V 写入预分片的本地缓存。较长的高分辨率潜在视频序列跨 rank 分割进行去噪，并通过 Ulysses 通信汇集，而短得多的音频潜在序列则不做序列分片直接生成。在这种划分下，think 器的语言/状态计算仅以 K/V 条件形式抵达执行器，因此无需在执行器组内传输单独的语言序列。这使额外的硬件集中用于视觉生成，同时保持了紧凑的 think-执行器边界，在计入 350 毫秒双向网络预算时，总远程交互延迟约为 550 毫秒。

## Abstract
We present Wan-Streamer v0.2, a latency-preserving upgrade of the native-streaming, end-to-end audio-visual interaction model. v0.2 keeps the v0.1 modeling formulation, but raises the interactive output stream from 192x336 to 640x368 while preserving approximately 200 ms model-side signal-to-signal latency at 25 FPS. The higher-resolution stream supports scene-grounded mid-shot agents whose posture, gaze, hands, nearby objects, and local scene layout remain legible during real-time conversation. To support the larger visual stream without adding user-visible delay, v0.2 keeps the thinker as a single-GPU low-latency path for streaming perception, the short language/state Transformer pass that builds the generation cache, and final decoding. The performer becomes a multi-GPU Ulysses-style context-parallel group for the expensive next-unit latent generation. Each performer rank writes incoming K/V into a pre-sharded local cache. The long high-resolution latent video sequence is split across ranks for denoising and gathered through Ulysses communication, while the much shorter audio latent sequence is generated without sequence sharding. In this split, the thinker's language/state computation reaches the performer only as K/V conditioning, so no separate language sequence has to be communicated inside the performer group. This concentrates additional hardware on visual generation while preserving the compact thinker-performer boundary, keeping total remote interaction latency at approximately 550 ms when a 350 ms bidirectional network budget is included.

## 1. 论文的核心问题与整体含义
- **背景与动机**：实时音视频交互模型（如 Wan-Streamer v0.1）需要在极低延迟（约 200 ms 模型侧）下生成同步的说话人脸视频。v0.1 的输出分辨率仅为 192×336，导致中景场景下代理的姿态、手部、物体等细节压缩严重，无法支撑“场景扎根”的自然交互。
- **核心问题**：如何在**不增加用户感知延迟**的前提下，将交互视频流的分辨率从 192×336 提升到 640×368（25 FPS），使中景代理的身体语言、注视、手部和场景上下文在实时对话中清晰可辨。
- **整体含义**：该升级将视觉交互范围从单人肖像视频通话拓展到更真实的“场景扎根中景”对话，提升沉浸感和表达能力，同时保持原系统的实时响应特性。

## 2. 论文提出的方法论
- **核心思想**：将部署模型拆分为 **Thinker（思考器）** 和 **Performer（执行器）** 两个角色，通过计算解耦与并行化吸收高分辨率带来的额外生成开销。
- **技术细节**：
  - **Thinker（单 GPU）**：负责流式感知、语言/状态更新、KV 缓存构建以及最终的音视频解码。它产生的语言/状态信息被封装为 **K/V 条件切片**，广播给执行器组，自身不参与昂贵的潜在生成。
  - **Performer（多 GPU Ulysses 上下文并行组）**：负责下一单元的潜在生成。长的高分辨率**视频潜在序列**跨 GPU rank 分割（序列并行），通过 Ulysses 风格的 all‑to‑all/gather 通信进行注意力计算和去噪，再汇集结果；短的**音频潜在序列**则在各 rank 上不分割地生成，避免不必要的通信开销。
  - **缓存设计**：每个执行器 rank 维护预分片的本地 KV 缓存，从 Thinker 接收的 K/V 切片直接写入对应分片，无需在执行器组内传输完整语言序列。
  - **延迟控制**：整个流水线嵌入在每 160 ms 的流式单元中：Thinker 处理当前用户输入，输出 K/V 切片和上一单元的解码响应；执行器并行更新缓存并生成下一单元潜在。模型侧信号‑信号延迟保持约 200 ms，总交互延迟约 550 ms（含 350 ms 网络预算）。
- **算法流程示意**（文字描述）：
  1. Thinker 编码 160 ms 用户音视频。
  2. Thinker 更新语言/状态 Transformer，构建 KV 缓存并广播兼容的 K/V 切片。
  3. 执行器各 rank 将 K/V 切片写入预分片本地缓存。
  4. 高分辨率视频潜在序列被均分给各 rank，执行 Ulysses 并行去噪（含 all‑to‑all 通信），同时音频潜在不分片直接生成。
  5. 汇聚后的视频潜在和音频潜在返回 Thinker，解码为下一 160 ms 的输出帧和音频。
